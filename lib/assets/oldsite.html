Metaprogramming makes me happy

I think I have mild OCD. My whole life I have been compelled to nitpick the small details of everything. During the 4th grade, I was tasked as the editor of a book compiling stories written by each of my classmates. I was told to take a look at the book and fix any glaring errors. I went far beyond the call of duty. By the time I was finished editing the 100 odd page book, I had normalized the line spacing of every paragraph, simplified every overly complicated sentence, corrected every typo and even in some cases revised content. My teacher was so impressed with my changes that she told the entire class what a great job I was doing. I was extremely proud of myself.

So far during my fellowship at the Flatiron School, I’ve been doing pretty well with labs. I’m having no trouble completing them quickly, with reasonably well written code. Before we reached the object oriented labs, I was starting to get a slightly bored. There wasn’t much difference between each of our code. There were only 3 states. Completed with good code, completed with bad code, or incomplete. It was pretty obvious viewing pull requests which solutions were the right way to do something and which were the wrong way to do something. The object oriented labs changed that paradigm, bringing new elements to our code. Finally I was able to not only make code that worked, but I was able to make design decisions that weighed multiple factors to decide what I believed was the best way to go about solving the problem. Here is a list of some factors to consider.

  • DRYness (Don’t Repeat Yourself)
  • Complexity (Simple is better)
  • Readability
  • Future proofing

In general, I tended to favor the directions of DRY and future proofing. I abstracted, divided and metaprogrammed as much as I possibly could, which made my code more modular and maintainable, at the cost of being more complex and less understandable. In my experience outside of my comfort zone, I really fell in love with metaprogramming.

What is Metaprogramming?

Metaprogramming is the writing of computer programs with the ability to treat programs as their data. It means that a program could be designed to read, generate, analyse or transform other programs, and even modify itself while running.1 In some cases, this allows programmers to minimize the number of lines of code to express a solution (hence reducing development time)[citation needed], or it gives programs greater flexibility to efficiently handle new situations without recompilation. Wikipedia

So far everyone in the class has seen or used a few different metaprogramming functions, possibly without knowing it. (Although Blake has stressed it over and over again) The attr_accessor method defines a getter and setter method for each argument passed to it. Below is some ruby code that generally approximates the C code that the attr_accessor method uses.

def attr_accessor(*args)
    args.each do | arg |
        # Getter
        define_method(arg.to_sym) do
            instance_variable_get("@" + arg)
        end
        # Setter
        define_method"#{arg}=".to_sym) do | val |
            instance_variable_set("@" + arg, val)
        end
    end
end

Something really neat about knowing how this works is that we can create our own slightly modified accessor methods for our own use.

In my oo-person lab, I was able to create my own accessor method that modifies the setter method which constrains my variable between 0 and 10.

def self.zerotenacc(*args)
    args.each do |arg|
        define_method(arg.to_sym) do
            instance_variable_get("@"+arg.to_s)
        end
        define_method("#{arg}=".to_sym) do | val |
            val = 10 if val > 10
            val = 0 if val < 0
            instance_variable_set("@"+arg.to_s, val)
        end
    end
    zerotenacc :happiness, :hygiene

In my playlister-cli lab, I recognized that my Artist, Genre, and Song classes each shared similar behaviors. In addition to its unique behaviors, each class needed to do the following:

  • Give each created object a name accessible by the name and name= methods.
  • Keep track of each object created by the class.
  • Count the number of objects spawned by itself.
  • Reset all created objects.
  • Search each object created by the class by name.
  • Create a new object and set its name.

Instead of defining each of these methods multiple times and repeating myself, I created a module which I used as a mixin to get my shared behavior into the class. Now, instead of having to repeat all of these behaviors for each class, all I have to do is call include KeepTrackOfNamedThing in each class.

module KeepTrackOfNamedThing
  def self.included base
    base.send :include, InstanceMethods
    base.extend ClassMethods
    base.instance_variable_set :@instance, []
  end

  module InstanceMethods
    def starter
        self.class.instance << self
    end

    attr_accessor :name
  end

  module ClassMethods
    def all
        @instance
    end
    def reset_all
        @instance = []
    end
    def count
        @instance.count
    end

    def find_by_name(name)
        i = @instance.index { | instance | instance.name == name }
        @instance[i] unless i.nil?
    end

    def create_by_name(name)
        obj = self.new
        obj.name = name
        obj
    end

    def instance
        @instance
    end
    def instance=(i)
        @instance = i
    end
  end
end

In the process of doing this, I learned some interesting tricks for working with modules and mixins. If you include a module in a class, your class receives all of the instance methods of the module as instance methods of the class. When you extend a module in a class, your class receives all of the instance methods of the module as class methods of the class.

I learned a cool metaprogramming trick to allow you to mixin both class methods and instance methods at the same time. You have to create two submodules inside of your module, one for the instance methods you want to inherit, and one for the class methods you want to inherit. Then, you use the included hook which Ruby calls after you include your module, and some metaprogramming to include your InstanceMethods submodule and to extend your ClassMethods submodule. Here is some boilerplate code:

module Module
  def self.included base
    base.send :include, InstanceMethods
    base.extend ClassMethods
  end

  module InstanceMethods
  end

  module ClassMethods
  end
end

In conclusion, metaprogramming is quite possibly the most interesting concept I have come across in programming to date. Ruby’s metaprogramming features cement the language in my mind as a great option for writing concise code.

Assignment is not that Simple

In my NYC Pigeon Organizer I was reminded of some of the trickiness of assignment and learned some new things about it in Ruby.

Assignment is the first operation every programmer learns in every language. The equals sign character is ubiquitous in programming for assigning an object to a namespace. I think most of us new programmers see assignment and immediately feel comfortable with it, but do we really understand exactly what is going on? In my case, not exactly.

a = "Hello World!"
b = a

This code causes b and a to both point to the same place in memory, our “Hello World!” string. We can verify that with the .object_id method.

a.object_id
=> 70241250209600
b.object_id
=> 70241250209600

Now if we change a, our pointers diverge.

a = "Goodbye World!"
a.object_id
=> 70241250146840
b
=> "Hello World!"

This is pretty logical. a and b used to point to the same value, but after a = "Goodbye World!" was executed, a was reassigned to a new value and b kept its old value.

When we start using mutator methods, or methods that change variables, things start behaving very differently.

a = "Foo"
b = a
b << "bar"
a
=> "Foobar"

This can be quite tricky. If we modify b using a mutator method, we actually change the data stored in memory at the point that both a and b are located.

a = "Foo"
b = a
b.object_id
=> 70241250098220
b << "bar"
a
=> "Foobar"
a.object_id
=> 70241250098220

The same happens with every bang method we execute. All existing pointers to the modified object will continue to exist.

This is a really interesting idea, so how else can we work around this?

This brings in 2 new methods. The .dup method and the .clone method. .dup and .clone perform similar tasks. They allow us to assign our variables to a copy of an object, so that when the original object is modified, the duplicated associations aren’t modified.

a = "foo"
=> "foo" 
b = a.clone
=> "foo" 
a << "bar"
=> "foobar" 
b
=> "foo"

Another interesting method is .freeze. This method locks an object for modification. A frozen object throws an exception when a modification to it is attempted.

b = "qwerty"
a = b
a.freeze
b << "asdf"
RuntimeError: can't modify frozen String

As you can see, freeze affects the object, not the pointer or variable.

In my NYC Pigeon Organizer, I wanted to make a template hash that would serve as a default value for every value in my pigeon hash. It looked something like this:

template = {}
data.each do |key, value|
    template[key] = []
end

When executed, my template hash looked like:

{:color=>[], :gender=>[], :lives=>[]}

When I assigned one of my birds this template, whenever I tried to shovel << a value into one of the bird’s attributes the entire template would get updated. As a quick fix, I tried to use the .dup method to make a copy of the template for each pigeon. After running the code, absolutely nothing changed and I couldn’t for the life of me figure out why. Why was my copy of the hash still pointing to the same places?

As is turns out, even though I duplicated the hash itself, all of its inner namespaces were still being associated with the same objects that the template hash was. With some Googling, I learned that what I really needed was called a “deep copy”, which is a core functionality of Python but not of Ruby. My first thought was to turn my template maker into a function, so every time the function was called, a new template would be created and the return value passed to where I needed it. This is a simple solution, but it runs into the problem that every time I want to add a template to my hash, I have to run a function that iterates through a large dataset.

With some more googling, I was able to find a super easy deep copy method which is somewhere in the middle for speed.

def ezdeepcopy(object)
    Marshal.load(Marshal.dump(object))
end

This function will deep copy our template hash by converting it into a binary stream and then converting it back into Ruby. This will let me copy reuse the inner hash without modifying the original template. While not as fast as writing a custom deep copy method for our data structure, this trick is fast enough for most uses.

Finder Changes for Developers

At the Flatiron School we use Macs. As far as I know every student in the class uses a borrowed Macbook Air from The City of New York. Before starting at Flatiron, the last time I used a Mac was in the mid 2000’s. My elementary school PowerPC (not Intel) Macbook ran OSX version 10.4 “Tiger”. The only reason I was able to determine which version it was is they added the Dashboard in that version and my 10 year old brain thought it would be important that I remembered that. Since then much has changed in the Apple world (including Apple completely changing the world with the iPhone). I feel like its a pretty stellar time to remind myself how to use a Mac and hopefully learn some new tricks to make my productivity a bit better.

Screenshots

I started writing the rest of this post without knowing how to take a screenshot on a Mac. I’m not sure how I would ever figure out how to do that if Google didn’t exist.

  • Command + Shift + 3 Full Screenshot
  • Command + Shift + 4 Interactive Screenshot

Finder Sidebar

Finder is Mac’s file manager. Its fairly similar to Windows Explorer and Nautilus (for all of you Linux users), but being a product made by Apple, Finder is designed for a group of people that use technology completely differently than everyone else. Finder’s default settings make it completely useless to a developer. Lets fix it. By going to Finder > Preferences or pressing the Command + , keybind, we can access the Finder preferences. Under the Sidebar tab, we can add our home folder. While we are here, we can also remove some of these completely useless options.

Let’s also set Finder to default to our home directory when we open a new window. In order to add our document root to the menu, we can press Command + Up until we see our hard drive icon and drag that to the list as well. Lets also drag our development folder into the list as well. Looks great!

New Terminal at Folder in Context Menu System Preferences > Keyboard > Shortcuts > Services Under Files and Folders: Enable New Terminal at Folder.

Now you can Control + Click any folder and press New Terminal at Folder. I need to sleep now. Maybe I can add some more of these another time.
